# LED 控制页状态边界优化设计

> 目标：收敛 `ControlPage` 的本地状态，让设备效果、亮度和错误状态只由 provider 维护，页面只保留纯 UI 状态。

## 背景

当前 `lib/features/control/control_page.dart` 同时维护了页面本地状态（`_currentEffect`、`_currentBrightness`）和 provider 状态。页面在发出控制命令后，会手动更新本地状态，再依赖 provider 执行网络请求。这会导致两个问题：

1. **状态来源不唯一**：页面显示值和真实设备状态可能不同步。
2. **后续扩展困难**：设备切换、状态刷新、失败回滚时，需要同时处理本地状态和 provider 状态。

这次优化只处理 `ControlPage` 的状态边界，不扩展到设备管理页或 WiFi 配置页。

## 目标

- 删除 `ControlPage` 中与设备业务相关的本地状态维护。
- 让页面直接以 `deviceProvider` / `lEDControlProvider` 作为状态来源。
- 保留页面自身的纯 UI 状态，例如当前 tab。
- 保持现有功能不变：灯效切换、亮度设置、状态刷新、错误提示都继续可用。

## 非目标

- 不重构 `device_manager` 页面。
- 不改动 UDP 协议或存储层。
- 不重新设计 provider 的数据模型。
- 不增加新的业务能力。

## 方案

### 方案 A：只收敛 `ControlPage` 本地状态（推荐）

将 `_currentEffect` 和 `_currentBrightness` 从 `ControlPage` 中移除，页面直接读取 provider 返回的控制状态。用户操作仍然通过 `lEDControlProvider` 发命令，成功后由 provider 更新状态，页面只负责重绘。

**优点**
- 改动范围最小。
- 能立刻消除页面与 provider 的双写问题。
- 容易验证，适合当前阶段。

**缺点**
- `ControlState` 仍然比较轻量，暂时不保存当前 effect/brightness 的显式值，需要依赖已有状态或后续扩展。

### 方案 B：扩大为控制状态模型重构

把 `ControlState` 扩展为更明确的业务状态模型，显式保存当前 effect、brightness、是否已同步等字段，再让 `ControlPage` 完全只读。

**优点**
- 状态表达更完整。
- 后续做设备切换、刷新、恢复更自然。

**缺点**
- 改动更大。
- 需要同步调整更多测试和 provider 行为。

### 方案 C：连页面拆分一起做

顺便把 `ControlPage` 拆成更小的组件，每个组件分别从 provider 取自己需要的状态。

**优点**
- 可维护性最好。
- 组件边界更清楚。

**缺点**
- 范围已经接近一次页面重构。
- 与当前“先修状态边界”的目标相比，收益不成比例。

## 选型

采用 **方案 A**。

理由：当前问题的核心是状态边界不清，而不是结构粒度不足。先把 `ControlPage` 的本地状态收敛掉，能最快验证单向数据流是否足够支撑当前功能；如果后续发现控制状态模型仍然不够，再升级到方案 B。

## 设计细节

### 1. `ControlPage` 的职责

`ControlPage` 只保留三类职责：

- 选择当前 tab。
- 根据 provider 状态渲染效果页、设置页和设备页。
- 将用户动作转发给 `lEDControlProvider`。

`ControlPage` 不再保存灯效和亮度的业务状态。

### 2. 状态来源

- 设备列表来源于 `deviceProvider`。
- 灯效命令、亮度命令、状态刷新由 `lEDControlProvider` 处理。
- 错误状态由 provider 负责记录，页面只做展示。
- 页面层只允许保留与导航/展示有关的局部状态。

### 3. 页面展示规则

- 如果没有设备，显示空态。
- 如果设备加载中，显示加载态。
- 如果设备列表加载失败，显示错误态。
- 当用户触发效果切换或亮度调节时，页面不自行保存目标值，改由 provider 的返回结果驱动 UI 更新。

### 4. 错误处理

- 网络请求失败时，`lEDControlProvider` 继续保留错误信息。
- `ControlPage` 仅负责把错误提示展示给用户。
- 空态、加载态、错误态保持与现有交互一致，不引入新交互。

## 测试策略

### 单元/组件层

- 验证 `ControlPage` 不再依赖本地 effect/brightness 状态。
- 验证在无设备、加载中、错误时页面仍能正确渲染。
- 验证效果切换和亮度调节仍然通过 provider 发出命令。

### 回归检查

- 启动应用后能正常进入控制页。
- 切换 tab 不影响控制页展示。
- 设备列表为空时不崩溃。
- 现有 widget test 继续通过。

## 迁移步骤

1. 从 `ControlPage` 中删除 `_currentEffect` 和 `_currentBrightness`。
2. 让页面从 provider 读取当前状态展示。
3. 保留 tab 选择逻辑不变。
4. 更新相关 widget test，使其匹配新的状态流向。
5. 运行测试，确认控制页行为未回退。

## 风险与缓解

- **风险：provider 当前状态过于轻量，页面展示信息不足。**
  - 缓解：先保持方案 A，必要时再升级为方案 B。

- **风险：页面依赖本地状态的代码被清理后，部分交互看起来没有立即变化。**
  - 缓解：用 provider 作为唯一状态源，避免双写后状态不一致。

## 结论

这次优化应优先收敛 `ControlPage` 的状态边界，确保业务状态只有一个来源。方案 A 能以最小成本解决当前主要问题，并为后续设备切换和状态同步打好基础。
